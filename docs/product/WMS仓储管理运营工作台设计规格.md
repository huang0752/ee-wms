# WMS 仓储管理运营工作台设计规格

版本：v1.0  
状态：设计中  
适用范围：`/module-wms` 仓储管理菜单下的 Web 管理端页面、WMS 业务接口展示契约、试用数据重建口径。  
约束：本规格先定义正式设计，不改代码实现；后续实现可清理现有测试数据，不要求兼容旧试用数据。

## 1. 背景与问题

当前仓储管理菜单已经按业务域拆成运营看板、基础资料、入库作业、出库作业、库存作业、分析追溯、试用工具，但除仓储驾驶舱外，大多数页面仍是普通后台 CRUD 形态：

1. 列表字段偏少，常见展示为单号、若干 ID、状态和操作。
2. 表格只展示数据，没有形成作业优先级、风险、进度和业务闭环。
3. 明细使用弹窗表格，无法承载单据主信息、明细、流程、库存影响、集成信息和操作记录。
4. 后端已预留部分业务字段，如来源、外部单号、同步状态、流程实例、库存桶、批次、库位规则等，但前端没有充分展示。
5. 菜单叫“仓储管理”，实际体验更像“表格集合”，不够像电工装备行业 WMS 产品。

## 2. 设计目标

把仓储管理从“表格菜单”升级为“仓储运营工作台体系”。

核心目标：

1. 每个业务页面必须先表达业务状态，再展示列表。
2. 表格保留，但升级为高密度作业表格，不做纯大屏化。
3. 单据详情统一使用右侧抽屉承载，不再依赖小弹窗展示明细。
4. 关键字段从 ID 型展示升级为业务型展示，优先展示编码、名称、状态、数量、时间、来源和风险。
5. 列表、详情、表单、状态、图表尽量复用现有 `fa-*` 组件和 Element Plus，不新造基础 UI 框架。
6. 允许清理并重建现有测试数据，正式字段和枚举以本规格为准，不为旧测试数据做兼容。

## 3. 总体信息架构

仓储管理菜单保持现有分组，但页面形态升级：

| 菜单 | 页面形态 | 目标 |
|---|---|---|
| 运营看板 / 仓储驾驶舱 | 综合驾驶舱 | 展示库存、任务、预警、流水和智能摘要 |
| 基础资料 / 仓储基础 | 主数据工作台 | 维护仓库、库区、库位、物料、供应商、客户、条码规则 |
| 入库作业 / 到货与检验 | 收货 IQC 工作台 | 串联采购到货、收货、检验、待入库 |
| 入库作业 / 入库管理 | 入库上架工作台 | 从检验生成入库，确认库存落账 |
| 出库作业 / 出库管理 | 销售出库工作台 | 锁库、拣货、复核、出库确认 |
| 出库作业 / 生产领料 | 生产领料工作台 | 按 MES 工单组织领料闭环 |
| 库存作业 / 库存管理 | 实时库存工作台 | 库存余额、库存桶、流水、冻结、锁库 |
| 库存作业 / 调拨管理 | 移库调拨工作台 | 调出、调入、确认、流水联动 |
| 库存作业 / 盘点管理 | 库存盘点工作台 | 账实差异、审核、调账 |
| 分析追溯 / 预警与分析 | 库存风险工作台 | 安全库存、缺料、呆滞、效期、处理建议 |
| 分析追溯 / 仓储追溯 | 批次追溯工作台 | 正向/反向追溯、流水和关系链 |
| 试用工具 / 试用数据 | 试用数据生成器 | 清理重建、生成行业化数据、质量报告 |

## 4. 页面统一布局规格

除驾驶舱和试用数据工具外，业务页统一采用以下结构：

```text
页面头部
  - 英文业务域标识
  - 页面标题
  - 业务说明
  - 右侧主操作：刷新、新建、导入导出、智能建议等

业务指标条
  - 3 到 5 个指标卡
  - 指标必须来自当前页面业务对象，不放泛泛 KPI

筛选工具条
  - 单号/编码/批次搜索
  - 状态筛选
  - 业务范围筛选，如仓库、物料、工单、预警类型

作业表格
  - 高密度表格
  - 状态标签、进度条、来源同步、时间节点
  - 操作列固定右侧

详情抽屉
  - 基本信息
  - 明细行
  - 流程轨迹
  - 库存影响
  - 集成信息
  - 操作记录或备注
```

## 5. 组件复用规范

优先复用：

| 用途 | 组件 |
|---|---|
| 页面壳 | `ElCard` + 现有主题变量；可沉淀 `WmsWorkbenchShell` |
| 指标卡 | `fa-stats-card` 或 WMS 专用薄封装 |
| 表格 | `fa-table` 或 `ElTable`，后续统一迁移到 `fa-table` |
| 状态 | `fa-status-tag` 或 WMS 状态封装 |
| 详情 | `fa-drawer` + `fa-descriptions` + `ElTabs` |
| 流程 | `ElSteps`、`fa-timeline-list-card` |
| 趋势图 | `fa-line-chart`、`fa-bar-chart`、`fa-ring-chart` |
| 图标 | `FaSvgIcon` + Remix/Iconify |

禁止：

1. 每个页面重复写一套状态颜色、状态中文、进度映射。
2. 在业务页面手写大量孤立样式，除非该样式确实是 WMS 业务布局。
3. 用纯 ID 字段作为主要展示信息，除非对应对象尚未接入名称查询。
4. 把所有信息塞进横向超长表格。

## 7. 页面详细规格

### 7.1 仓储驾驶舱

定位：仓储管理的第一入口。

必须展示：

1. 库存总量、可用库存、待检库存、锁定库存、冻结库存。
2. 待收货、待检验、待入库、待发料、待盘点。
3. 未关闭预警数量与风险等级。
4. 最新库存流水。
5. 库存结构、出入库趋势、预警分布。
6. 智能摘要，但事实数据必须来自规则和数据库，AI 只做解释。

后续可扩展大屏页，但普通驾驶舱优先服务日常工作。

### 7.2 仓储基础

页面目标：主数据不是简单字典，而是仓储网络、物料规则和条码策略的配置中心。

资源 Tab：

1. 仓库
2. 库区
3. 库位
4. 物料
5. 供应商
6. 客户
7. 条码规则

列表字段：

| 资源 | 关键字段 |
|---|---|
| 仓库 | 编码、名称、类型、负责人、部门、状态、备注 |
| 库区 | 编码、名称、所属仓库、用途、状态 |
| 库位 | 编码、名称、所属仓库、库区、容量、承重、类别限制、混放规则、状态 |
| 物料 | 编码、名称、规格、单位、类别、批次管理、SN 管理、安全库存、状态 |
| 供应商 | 编码、名称、联系人、电话、邮箱、地址、状态 |
| 客户 | 编码、名称、联系人、电话、邮箱、地址、状态 |
| 条码规则 | 编码、名称、对象类型、前缀、分段策略、状态 |

建议补字段：

| 对象 | 字段 |
|---|---|
| 库位 | `load_capacity`、`temperature_zone`、`abc_class`、`is_pickable` |
| 物料 | `storage_condition`、`shelf_life_days`、`min_pack_qty`、`critical_flag` |
| 条码规则 | `date_format`、`serial_length`、`sample_code` |

5. 来源/同步

4. 锁定数量

流程：


## 8. 字段与接口增强规格

### 8.1 列表接口增强

后续列表接口不应只返回外键 ID，建议增加展示快照字段：

| 对象 | 建议返回 |
|---|---|
| 单据 | `warehouse_code`、`warehouse_name`、`source_label`、`sync_label` |
| 明细 | `material_code`、`material_name`、`material_spec`、`unit` |
| 库位 | `location_code`、`location_name`、`zone_name` |
| 供应商/客户 | `supplier_code/name`、`customer_code/name` |

这些字段可以通过查询时 join 或服务层组装，避免前端再按 ID 二次请求。

### 8.2 详情接口增强

建议每个单据提供详情聚合接口：

```text
GET /wms/arrival/{id}/detail
GET /wms/inbound/{id}/detail
GET /wms/outbound/{id}/detail
GET /wms/issue/{id}/detail
GET /wms/transfer/{id}/detail
GET /wms/check/{id}/detail
```

返回结构：

```json
{
  "header": {},
  "lines": [],
  "flows": [],
  "trace_links": [],
  "warnings": [],
  "operation_logs": []
}
```

### 8.3 统计接口增强

每个工作台可以有轻量统计接口：

```text
GET /wms/arrival/metrics
GET /wms/outbound/metrics
GET /wms/stock/metrics
```

初期可前端基于当前页计算，正式版本建议后端按租户全量计算。

## 9. 数据重建规格

## 10. 验收标准

### 10.1 视觉与交互

1. 仓储管理下所有业务页不再是单纯“搜索 + 表格 + 弹窗明细”。
2. 每个业务页都有页面说明、指标卡、筛选工具条、作业表格和详情抽屉。
3. 状态必须使用中文状态标签，不直接暴露英文枚举。
4. 流程型页面必须展示进度。
5. 数量型明细必须展示计划/实收/合格/锁定/出库/差异等业务数量。
6. 移动宽度下页面不发生文字重叠。

### 10.2 业务完整性

1. 到货可收货并生成检验任务。
2. 检验可判定合格/不合格。
3. 入库确认后库存余额和流水正确。
4. 出库/领料按锁库、拣货、复核、确认推进。
5. 调拨确认后产生调出和调入流水。
6. 盘点审核后产生调整流水。
7. 预警能扫描、展示、关闭。
8. 追溯能按批次查到相关流水和链路。

# WMS 框架依赖现状评估

本文基于 `ee-wms` 当前代码状态评估原始框架问题清单，避免后续 WMS 开发重复补已经落入框架的能力。

## 1. 当前结论

WMS 可以开始进入产品技术规划和第一阶段开发，不需要等待框架从零补齐基础能力。当前优先级应从“修框架基础问题”转为“定义 WMS 领域模型、库存账规则、菜单权限、试用数据生成器和产品 assembly”。

仍建议把可复用能力优先沉淀到框架：业务任务、AI 调用审计、行业词库、通知待办、代码生成模板、上传分类和插件生命周期治理。WMS V1 前台不直接展示这些框架管理入口，业务侧需要时应在 `module_wms` 下接入。

## 3. WMS 开发前置清单

1. 维护 WMS 产品 assembly：后端保留 `module_ai`、`module_task`、`module_generator`，前台启用系统、平台、多租户工作台和 `module-wms` 等运行入口，隐藏 AI/任务/代码生成、通用仪表盘、支付、套餐、订单、发票、插件市场、公告和工单入口。
2. 新增 WMS seed pack：菜单、权限、推荐角色、部门样例、基础字典、行业词库。
3. 定义 WMS 数据模型：先冻结仓库、库区、库位、物料、供应商、客户、批次库存、库存余额、库存流水。
4. 定义库存账规则：所有库存数量变化先写流水，再更新余额；待检、冻结、锁定库存不可被推荐出库。
5. 定义试用数据生成器：使用 `BusinessTask` 记录进度，所有演示数据带 `is_demo` 和 `demo_batch_id`。
6. 定义前端页面约束：所有列表页用 `FaTable` 组合，所有状态展示用统一枚举和 tag。
